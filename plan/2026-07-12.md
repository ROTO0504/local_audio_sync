## 1. 背景と目的

現状のアプリは「LAN 内の複数デバイスの内部音声を **Windows の Hub** に集約して再生する」構成で安定動作している。今回、以下の拡張要望を受けた:

1. **Hub(集約・再生側)を Windows 以外でも動かせるようにする** — iPhone / Android / Mac / Windows のどの端末でも受信・再生役になれる
2. 複数クライアント接続と、**クライアントごとの音量調節 UI**(既存機能の強化)
3. **再接続の堅牢化とデバイス ID の永続化** — 再起動しても同じデバイスとして認識され、音量設定などが復元される
4. **Hub からクライアントへのリモート制御** — 配信の一時停止 / 再開 / 停止を Hub の画面から操作できる
5. (できれば)**ネットワーク越え** — ブロードキャストが届かないセグメントや WAN 経由でも接続できる

## 2. 用語定義

会話やドキュメントで「ホスト / クライアント」の指す側が揺れやすいので、本計画では既存コードの用語に統一する:

| 用語 | 役割 | 別名 |
| --- | --- | --- |
| **Hub** | 音声を集約して 1 台で再生する側(受信側) | ホスト |
| **Client** | 自端末の内部音声を Hub に送る側(送信側) | 送信クライアント |

「再生停止のリモート制御」は **Client の配信ストリームを Hub から止める / 再開する** ことを指す。Client 端末上の音楽アプリ(Spotify 等)自体の再生操作は OS 制約によりスコープ外(§10)。

## 5. アーキテクチャ全景(変更後)

```text
┌─────────────────────────────────────────────────────────────┐
│  Client(送信側)— 全 OS(変更なし + 制御受信を追加)          │
│                                                             │
│  内部音声キャプチャ → PCM16/48kHz/20ms → Opus 128kbps       │
│  → UDP ユニキャスト(7777)                                  │
│                                                             │
│  ＋ CMD 受信(PAUSE / RESUME / STOP)→ 送信ゲート制御        │
│  ＋ 永続デバイス ID(初回生成 → shared_preferences)          │
└─────────────────────────────────────────────────────────────┘
                     ↑↓(制御は双方向)
┌─────────────────────────────────────────────────────────────┐
│  Hub(集約・再生側)— ★全 OS 対応化                          │
│                                                             │
│  UDP 受信 → JitterBuffer → Opus デコード → 個別音量          │
│  → miniaudio ミキサー(FFI プラグイン化: Win/mac/iOS/Android)│
│  → スピーカー出力                                            │
│                                                             │
│  ＋ HubController(UI から分離、per-device 設定永続化)       │
│  ＋ CMD 送信(ACK + 再送)                                    │
└─────────────────────────────────────────────────────────────┘

ディスカバリー(デュアルスタック):
  - mDNS/Bonjour: Hub がサービス `_lasync._udp` を公開 ★新規(iOS Hub の必須要件)
  - UDP ビーコン(9999): 従来方式を併送(旧バージョン互換 + 保険)
  - 手動 IP:port 指定 ★新規(ブロードキャスト不達環境 / VPN / WAN 用)
```

## 6. プロトコル v2 仕様

### 6.1 制御メッセージ(UDP 7777、テキスト)

既存メッセージは変更せず、以下を追加する。**旧クライアント / 旧 Hub との混在でも壊れない**(未知メッセージは無視される)。

| メッセージ | 方向 | 形式 | 説明 |
| --- | --- | --- | --- |
| HELLO2 | C → H | `HELLO2:{name}:{uuid}:{platform}:{proto}` | platform = `ios/android/macos/windows`、proto = `2`。旧 `HELLO:` も引き続き受理 |
| PONG | H → C | `PONG:{clientId}` | PING への応答。クライアントはソケットレベルで Hub 生存を確認できる(ビーコン喪失検出の補完) |
| CMD | H → C | `CMD:{clientId}:{cmdSeq}:{PAUSE\|RESUME\|STOP}` | リモート制御。cmdSeq は Hub 管理の単調増加値(重複実行防止) |
| CMDACK | C → H | `CMDACK:{clientId}:{cmdSeq}` | CMD の到達確認。Hub は ACK が来るまで 500ms 間隔・最大 5 回再送 |

- CMD の送り先は HELLO 受信時に記録済みの `(ip, port)`(`ClientInfo.port`)。クライアントは既に ACKHELLO / RESYNC を同一ソケットで受けているため、受信経路の追加実装は小さい。
- クライアントは同一 cmdSeq の CMD を再度受けたら **ACK のみ返して再実行しない**。

### 6.2 ディスカバリー v2

| 方式 | 実装 | 用途 |
| --- | --- | --- |
| mDNS / Bonjour | `bonsoir` パッケージ。サービス型 `_lasync._udp`、TXT に `name` / `hubId` / `proto` | 全 OS の Hub 公開。**iOS Hub はこれが唯一の手段**(Bonjour API 経由なら multicast entitlement 不要。`Info.plist` に `NSBonjourServices` + `NSLocalNetworkUsageDescription` を追加) |
| UDP ビーコン v2 | `LAHUB2:{ip}:{port}:{name}:{hubId}:{proto}` を従来の `LAHUB:` と交互送信 | Windows / Android / macOS Hub の従来互換 + mDNS が死んだ環境の保険 |
| 手動指定 | クライアント UI に IP:port 入力 + 接続履歴 | セグメント越え / VPN / WAN(フェーズ 5) |

- `hubId` = Hub 側の永続 UUID(新規)。クライアントは「前回接続した Hub」を記憶し、複数 Hub が見つかった場合に優先接続する。
- クライアントの探索は mDNS ブラウズと ビーコン受信を並走させ、同一 `hubId`(または ip:port)は重複排除する。

### 6.3 音声パケット

変更なし(マジック `0xA1A2` / clientId / seq / Opus)。PAUSE は「送信側が音声パケットを止める」だけで、プロトコル変更を伴わない。

## 7. フェーズ別実装計画

> フェーズ 1 → 2 は直列(2 以降は 1 の HubController 抽出に依存)。フェーズ 2(ネイティブ)と 3〜4(Dart/UI)は独立性が高く、順序の入れ替え・並行が可能。

### フェーズ 1: 基盤 — デバイス ID 永続化・プロトコル v2・Hub ロジック抽出

**目的**: R3 の土台と、以降すべてのフェーズが乗る配線を先に固める。

**サブタスク**:

1. `DeviceIdentityService` 新設: 初回起動時に UUID v4 を生成して `shared_preferences` に保存、以降は同一 ID を返す。Hub 用 `hubId` も同様に永続化。`client_screen.dart:38` の都度生成を置換。
2. `HELLO2`(platform / proto 付き)送信 + Hub 側の HELLO / HELLO2 両対応パース。`ClientInfo` に `platform` / `protocolVersion` を追加。
3. `PING` に対する `PONG` 応答を Hub に実装。クライアント側は PONG 未受信の連続回数を再接続トリガに加える(既存のビーコン喪失検出は維持)。
4. `CMD` / `CMDACK` の送受信配線と再送キュー(500ms × 5 回)を `udp_receiver_service.dart` / `udp_sender_service.dart` に実装。**この段階では受信してもログのみ**(実動作はフェーズ 4)。
5. `hub_screen.dart` に直書きされた Hub コアロジック(HELLO 処理、`_uuidToClientId`、stale 監視、ミキサー操作)を `lib/services/hub_controller.dart` へ抽出。UI は Riverpod 経由で状態を購読するだけにする。
6. ユニットテスト: HELLO2 パース(旧形式との互換含む)、CMD 再送 / 重複排除、PONG タイムアウト、HubController の接続ライフサイクル。

**コミット境界**:

- :sparkles: `HELLO2 と PONG をプロトコルに追加(旧形式互換)`

### フェーズ 2: Hub のクロスプラットフォーム化(R1、最重要)

**目的**: iPhone / Android / Mac でも Hub(集約・再生)になれるようにする。

**方針**: 既存の miniaudio ベース C++ ミキサー(`windows/audio_mixer_plugin/audio_mixer.cpp`)は miniaudio が全 OS 対応のため **ソースをほぼそのまま流用**し、Flutter FFI プラグインとして各 OS のビルドグルーを整備する。Dart でのミックス再実装(代替案)は既存資産を捨てるため採らない。

**サブタスク**:

1. C++ ソースを `packages/audio_mixer_ffi/src/` に移設し、FFI プラグインパッケージ化(`ffiPlugin: true`)。Windows は既存 CMake を移設(loopback API も同梱、既存動作を維持)。
2. **Android**: `CMakeLists.txt` + NDK ビルド(`libaudio_mixer.so`)。miniaudio の AAudio バックエンド。再生継続用フォアグラウンドサービス(type = `mediaPlayback`)を追加(既存のキャプチャ用 `AudioBroadcastService` とは別)。
3. **iOS / macOS**: podspec でソースを静的リンクし `DynamicLibrary.process()` で解決(libopus フレームワーク追加時と同じ方式)。iOS は `AudioSessionManager` に `.playback` カテゴリの Hub モードを追加し、`UIBackgroundModes: audio` でバックグラウンド再生を維持。macOS は `com.apple.security.network.server` entitlement の付与を確認。
4. `AudioMixerService` のライブラリロードを OS 分岐対応(`Platform.isWindows` ガードの撤廃)。
5. **mDNS ディスカバリー併設**(§6.2): `bonsoir` 導入。Hub 起動時にサービス公開 + 従来ビーコン併送、クライアントは両方を購読。iOS の `Info.plist` に `NSBonjourServices`(`_lasync._udp`)を追加。
6. ロール選択画面(`setup_screen.dart`)の制限を解除し、全 OS で「Hub(集約・再生)」を選択可能に。
7. 各 OS でのビルド確認(Windows はローカル、iOS / macOS / Android は CI +実機はユーザー依頼)。

**コミット境界**:
- :recycle: `ミキサー C++ を FFI プラグインパッケージ audio_mixer_ffi に移設`
- :sparkles: `Android で miniaudio ミキサーをビルドし Hub 再生に対応`
- :sparkles: `iOS/macOS で miniaudio ミキサーを静的リンクし Hub 再生に対応`
- :sparkles: `mDNS(Bonjour)による Hub 公開と探索を追加`
- :sparkles: `全プラットフォームで Hub ロールを選択可能に`
- :construction_worker: `CI に audio_mixer_ffi の各 OS ビルドを追加`

### フェーズ 4: リモート制御 — Hub → Client(R4)

**目的**: Hub の画面から各クライアントの配信を一時停止 / 再開 / 停止できるようにする。

**サブタスク**:

1. `ClientInfo` に `isPaused` を追加。HubController に `pauseClient(uuid)` / `resumeClient(uuid)` / `stopClient(uuid)` を実装(フェーズ 1 の CMD チャネルを使用)。
2. クライアント側の CMD 実動作:
   - **PAUSE**: キャプチャは維持したまま **UDP 送信をゲートで止める**(全 OS で一貫・即効。ミュートと違い帯域も止まる)
   - **RESUME**: 送信再開(再開時に seq 連続性が崩れるため、RESYNC と同じ経路で seq を 0 リセットしてから送出)
   - **STOP**: キャプチャ自体を停止。Android / macOS / Windows はキャプチャ API を停止。**iOS は Picker 制約により Extension を直接止められない**ため、UDS 経由で Extension に停止要求を送り `finishBroadcast` を呼ばせる(ベストエフォート。最低限、送信ゲートは閉じる)
3. クライアント UI: 「Hub により一時停止中」バナー表示と、ローカルからの再開ボタン(ローカル操作を優先するか Hub 操作を優先するかは「後勝ち」で統一)。
4. Hub UI: タイルに ⏸ / ▶ ボタン、ヘッダに「全員一時停止 / 全員再開」ボタン。CMDACK 未達(再送 5 回失敗)時はタイルにエラー表示。
5. 一時停止中も PING / PONG は継続(接続維持と区別するため)。
6. ユニットテスト: CMD 受信 → 送信ゲート状態遷移、cmdSeq 重複時の非再実行、RESUME 後の seq リセット。

**コミット境界**:
- :sparkles: `Hub からクライアントの配信を一時停止/再開/停止できるリモート制御を追加`
- :sparkles: `クライアントに Hub 操作による一時停止状態の UI を追加`
- :lipstick: `Hub 画面に全員一時停止/再開の一括操作を追加`
- :white_check_mark: `リモート制御の状態遷移テストを追加`

### フェーズ 5(オプション): ネットワーク越え(R5)

**目的**: ブロードキャスト / mDNS が届かない環境(別セグメント、VPN、WAN)でも接続できるようにする。

**サブタスク**:

1. クライアントに **手動接続 UI**: `IP:port` 入力 + 接続履歴(最近 5 件を `shared_preferences` に保存)。ディスカバリーと排他ではなく「手動接続中は探索を停止」する状態遷移を `client_state_provider` に追加。
2. JitterBuffer の**遅延プリセット**: 「LAN(低遅延、現行値)」/「WAN(安定重視、目標遅延とバッファ深さを拡大)」を接続時に選択可能に。パケロス率と再同期頻度をクライアント統計としてタイルに表示(任意)。
3. **VPN 運用ガイド**を README に追加: Tailscale / WireGuard で端末同士を同一仮想ネットワークに入れ、手動 IP 接続すれば WAN 越しに動作する(暗号化も VPN 層で担保される)。**平文 UDP を裸のインターネットに直接晒す運用は非推奨と明記**。
4. (将来検討・本計画では実装しない)NAT 自動越え: シグナリングサーバ + UDP ホールパンチング + DTLS/AEAD 暗号化。必要になった時点で別計画を起こす。

**コミット境界**:
- :sparkles: `Hub への手動 IP 接続と接続履歴を追加`
- :sparkles: `JitterBuffer に LAN/WAN 遅延プリセットを追加`
- :memo: `VPN 経由の WAN 利用ガイドを README に追加`

## 8. テスト方針

- **ユニットテスト**(既存 54 件 + 追加):
  - プロトコル v2: HELLO2 / PONG / CMD / CMDACK のパースと後方互換(旧 HELLO 受理)
  - CMD 再送キュー: ACK 到達で停止、5 回失敗でエラー、cmdSeq 重複の非再実行
  - HubController: 接続ライフサイクル、per-uuid 設定復元、inactive 遷移
  - 送信ゲート: PAUSE / RESUME / STOP の状態遷移、RESUME 後の seq リセット
  - JitterBuffer: 遅延プリセット別の挙動
- **ネイティブ層**: 各 OS のビルド通過を CI で確認。音出しの実機検証はユーザー依頼(チェックリストを README に用意)。
- **静的解析**: `flutter analyze` 警告ゼロを維持。

## 9. リスクと軽減策

| リスク | 軽減策 |
| --- | --- |
| iOS の UDP ブロードキャスト送信は multicast entitlement(Apple 申請制)が必要で、iOS Hub がビーコンを送れない | ディスカバリーを mDNS(Bonjour API は entitlement 不要)主体に。ビーコンは他 OS の互換用に併送 |
| `bonsoir` の Windows / 特定環境での mDNS 信頼性 | ビーコンとのデュアルスタックで冗長化。最悪でも手動 IP 接続(フェーズ 5)で救済 |
| miniaudio のモバイルバックエンド(AAudio / Core Audio)で音声割り込み(電話着信・デバイス切替)後に出力が止まる | AudioSession / AudioFocus の割り込み通知を受けてミキサーデバイスを再初期化する復帰処理を入れる |
| iOS Hub のバックグラウンド再生が OS に殺される | `UIBackgroundModes: audio` + 再生継続で維持(音楽アプリと同等)。無音時の維持は保証外と README に明記 |
| Broadcast Extension を Hub 側 STOP で完全停止できない(iOS の Picker 制約) | 送信ゲート停止を保証動作とし、Extension 停止はベストエフォート(UDS 経由の停止要求) |
| uint16 clientId が再接続で変わり、ミキサー内の状態と不整合 | 状態管理は常に永続 uuid をキーにし、clientId は「現在のセッションの転送用番号」に役割を限定(フェーズ 1 の HubController 抽出で整理) |
| WAN でのジッタ / パケロス増大 | JitterBuffer の WAN プリセット + 既存の RESYNC 機構。統計表示で可視化 |
| 平文 Opus を WAN に晒す盗聴リスク | VPN(Tailscale 等)経由を推奨手順として文書化。裸の WAN 直結は非推奨と明記 |
| FFI プラグイン化で Windows の既存 loopback(Client 機能)が壊れる | loopback API も同一パッケージに移設し、既存テスト(`windows_loopback_service_test.dart`)を維持・拡張 |

## 10. スコープ外(やらないこと)

| 項目 | 理由 |
| --- | --- |
| Client 端末上の音楽アプリ(Spotify / YouTube 等)自体の再生・停止操作 | iOS は他アプリのメディア制御 API がない。リモート制御の対象は「配信ストリーム」のみ(Android の MediaSession 経由の制御は将来検討課題) |
| DRM 保護コンテンツのキャプチャ | OS 仕様(従来どおり) |
| NAT 自動越え(ホールパンチング / 中継サーバ) | 規模が大きい。本計画では VPN 運用ガイドで代替し、設計のみ将来課題として残す |
| 1 クライアントから複数 Hub への同時配信 | ユースケース不明のため。1 Client → 1 Hub を維持 |
| Hub 音声の再配信(Hub → 別デバイスへの出力転送) | 今回の要望に含まれないため |
